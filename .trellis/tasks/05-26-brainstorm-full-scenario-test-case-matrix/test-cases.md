# OpenCode-CUI 全量业务测试用例文档

> 目标：用业务语言把 CUI 小程序、IM 单聊、IM 群聊在本地助手、远端助手、默认助手下的所有核心场景列清楚。  
> 当前口径：IM 单聊/群聊统一走“IM 业务统一入口”，旧 IM 入站接口只保留兼容 smoke，不作为主链路验收标准。

---

## 1. 阅读方式

这份文档分三层：

| 层 | 用来回答什么问题 |
| --- | --- |
| 覆盖矩阵 | 有没有漏掉入口、助手类型、用户动作、失败场景 |
| 详细用例 | 测试同学具体怎么测，什么算通过 |
| 最小验收套件 | 时间不够时，哪些用例必须先跑 |

每条详细用例尽量按下面格式写：

| 字段 | 含义 |
| --- | --- |
| 业务场景 | 用户或业务系统正在做什么 |
| 前置准备 | 测试前要准备哪些助手、会话、配置或模拟返回 |
| 操作步骤 | 测试同学具体执行什么 |
| 预期结果 | 用户、业务侧或页面应该看到什么 |
| 验收重点 | 最容易出问题、必须重点看的点 |
| 备注 | 必要时补充技术入口，但不作为主表达 |

## 2. 业务术语

| 术语 | 业务含义 |
| --- | --- |
| CUI / 小程序 | 用户在小程序里和助手聊天 |
| IM 单聊 | 用户在 IM 里和助手一对一聊天 |
| IM 群聊 | 用户在 IM 群里向助手提问或 @ 助手 |
| 本地助手 | 助手运行在本地插件/本地 agent 侧，需要在线 |
| 远端助手 | 助手运行在云端，不依赖本地 agent 在线 |
| 默认助手 | 用户没有明确选择助手时，系统按业务场景自动选中的助手 |
| 默认协议远端助手 | 远端助手使用普通云端消息格式 |
| 助手广场远端助手 | 远端助手使用助手广场消息格式 |
| 追问 | 助手向用户补充提问，例如“请选择一个方案” |
| 授权 | 助手执行操作前询问用户是否允许 |
| 重建会话 | 当前上下文异常、过长或用户主动重开时，系统重新建立干净上下文 |
| 跨实例 | 用户连接在一台服务上，但助手回复回到另一台服务上 |
| 云端主动推送 | 云端主动向 IM 发消息，不是用户先问后答 |

| 助手请求授权，用户选择 | 必测 | 必测 | 必测 |
| 用户或系统重建会话 | 必测 | 必测 | 必测 |
| 助手离线 | 本地助手必测 | 本地助手必测 | 本地助手必测 |

| 并发首条消息 | 可选 | 必测 | 必测 |

### 3.3 失败和边界场景

| 失败场景 | 必测原因 |
| --- | --- |
| 消息为空 | 防止空消息进入助手链路 |
| 发送人为空 | 防止系统用助手 owner 冒充真实用户 |
| 助手账号为空 | 防止找不到助手还创建会话 |
| 助手已删除 | 用户要看到明确不可用提示 |
| 助手状态未知 | 用户要看到无效助手提示 |
| 本地助手离线 | 本地助手需要在线，远端助手不应受影响 |
| 默认助手规则缺失 | 默认助手不能随机兜底 |
| 默认助手规则错误 | 错误配置不能污染正常会话 |
| 云端地址缺失 | 远端助手要给明确错误，不要无限重试 |
| 云端超时/返回错误 | 用户不能一直等待 |
| 跨实例投递 | 多实例部署下不能丢消息或重复回复 |
| 重复云端推送 | 同一条云端主动消息不能发两次 |

## 5. CUI 小程序详细用例

### CUI-L-01 小程序第一次问本地助手

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户打开小程序，选择本地助手 A，第一次发送“你好”。 |
| 前置准备 | 本地助手 A 在线；小程序能展示该助手；用户没有当前会话。 |
| 操作步骤 | 1. 打开小程序。2. 选择本地助手 A。3. 发送“你好”。4. 让助手返回一段文字并结束。 |
| 预期结果 | 页面立即显示用户消息；助手开始返回文字；回答结束后输入框恢复可用；刷新页面后历史仍能看到这轮对话。 |
| 验收重点 | 第一次消息不能在等待创建会话时丢失；不能创建多个重复会话。 |

### CUI-L-02 小程序继续问本地助手

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户在同一个小程序会话里继续问第二个问题。 |
| 前置准备 | 已完成 CUI-L-01，原会话仍存在。 |
| 操作步骤 | 在同一会话发送“继续帮我分析”。 |
| 预期结果 | 系统复用原会话；助手基于同一上下文回答；历史中追加第二轮对话。 |
| 验收重点 | 不能新开一个会话；不能把第二轮问题放错历史。 |

### CUI-L-03 小程序本地助手追问

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 本地助手需要用户补充选择。 |
| 前置准备 | 本地助手 A 能返回追问卡片。 |
| 操作步骤 | 1. 用户发送一个会触发追问的问题。2. 页面出现追问卡片。3. 用户选择答案。 |
| 预期结果 | 追问卡片展示清楚；用户提交后卡片状态更新；助手继续回答。 |
| 验收重点 | 回答必须回到原追问；如果追问来自子任务，不能落到主任务。 |

### CUI-L-04 小程序本地助手授权

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 本地助手执行操作前请求用户授权。 |
| 前置准备 | 本地助手 A 能返回授权卡片。 |
| 操作步骤 | 分别测试“允许一次”“始终允许”“拒绝”。 |
| 预期结果 | 三种选择都能提交；页面展示授权结果；助手按选择继续或停止。 |
| 验收重点 | 授权值不能串会话；非法授权值必须被拒绝。 |

### CUI-L-05 小程序本地助手离线

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户选择本地助手 B，但助手不在线。 |
| 前置准备 | 本地助手 B 处于离线状态。 |
| 操作步骤 | 小程序选择本地助手 B 并发送消息。 |
| 预期结果 | 用户看到明确的离线或不可用提示；页面恢复可输入；不会一直转圈。 |
| 验收重点 | 这类失败只影响本地助手，不应影响远端助手。 |

### CUI-RD-01 小程序问远端默认协议助手

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户选择远端默认协议助手，发送普通问题。 |
| 前置准备 | 远端默认协议助手配置正常；云端模拟服务能正常返回文本。 |
| 操作步骤 | 1. 选择远端默认协议助手。2. 发送“帮我总结一下”。3. 云端返回流式文本并结束。 |
| 预期结果 | 页面展示流式回答；结束后可继续输入；历史保存最终回答。 |
| 验收重点 | 远端助手不依赖本地助手在线；结束信号必须让页面恢复空闲。 |

### CUI-RD-02 小程序远端默认协议追问和授权

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 远端默认协议助手向用户追问或请求授权。 |
| 前置准备 | 云端能分别返回追问、授权请求。 |
| 操作步骤 | 1. 触发追问并提交答案。2. 触发授权并分别选择允许/拒绝。 |
| 预期结果 | 用户操作能回到云端；云端继续返回后续结果；页面状态正确。 |
| 验收重点 | 追问和授权不能被当成普通聊天消息。 |

### CUI-RD-03 小程序远端默认协议云端失败

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 远端默认协议助手云端不可用。 |
| 前置准备 | 云端模拟服务返回错误、超时或缺少配置。 |
| 操作步骤 | 用户发送一条消息。 |
| 预期结果 | 用户看到明确失败提示；页面恢复可操作；系统不反复重建会话。 |
| 验收重点 | 云端配置缺失不是上下文丢失，不能无限重试。 |

### CUI-AS-01 小程序问助手广场助手

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户选择助手广场来源的远端助手。 |
| 前置准备 | 助手广场云端模拟服务能返回文本、心跳、结束标记和未知事件。 |
| 操作步骤 | 发送普通问题，并让云端按助手广场格式返回。 |
| 预期结果 | 用户只看到真实回答；心跳、结束标记、未知事件不显示成脏消息；历史正确。 |
| 验收重点 | 助手广场协议不能用默认协议方式误解析。 |

### CUI-DEF-01 小程序默认助手命中

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 小程序某业务场景没有显式选择助手，系统自动使用默认助手。 |
| 前置准备 | 配置 CUI 默认助手规则。 |
| 操作步骤 | 不显式传入助手，创建会话并发送消息。 |
| 预期结果 | 系统选择正确默认助手；用户收到回复；历史可查。 |
| 验收重点 | 默认助手是按业务场景选的，不能随机兜底。 |

### CUI-DEF-02 小程序默认助手规则缺失

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 小程序需要默认助手，但没有配置规则。 |
| 前置准备 | 删除或停用 CUI 默认助手规则。 |
| 操作步骤 | 不显式选择助手，创建会话。 |
| 预期结果 | 系统返回明确失败；用户不会进入一个无法回复的空会话。 |
| 验收重点 | 不能静默创建异常会话。 |

### CUI-COM-01 小程序断线恢复

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户正在看助手回答时网络断开，然后重新进入。 |
| 前置准备 | 任意助手正在流式回答。 |
| 操作步骤 | 1. 回答过程中断开页面连接。2. 让助手继续返回。3. 用户重新进入会话。 |
| 预期结果 | 用户能看到最新回答；不会看到其他用户消息；最终状态正确。 |
| 验收重点 | 断线期间消息不能丢，也不能重复多遍。 |

### CUI-COM-02 小程序历史恢复

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户查看历史会话。 |
| 前置准备 | 已完成多轮普通回答、追问、授权。 |
| 操作步骤 | 刷新页面或重新进入历史会话。 |
| 预期结果 | 用户消息、助手回答、追问状态、授权状态顺序正确。 |
| 验收重点 | 实时看到的内容和历史恢复出的内容要一致。 |

### CUI-COM-03 小程序选中文本发送到 IM

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户在小程序里选中助手回答的一段文字，发送回当前 IM 会话。 |
| 前置准备 | 当前小程序会话有关联 IM 单聊或群聊。 |
| 操作步骤 | 选中一段文本，点击发送到 IM。 |
| 预期结果 | 文本发送到正确 IM 会话；空内容或超长内容被拦截。 |
| 验收重点 | 目标 IM 会话由后台当前会话判断，不能让前端随意指定错误目标。 |

## 6. IM 单聊详细用例

> IM 单聊主链路：用户在 IM 单聊中发消息，IM 业务系统通过统一入口转给 CUI 后端，助手回复再通过 IM 业务连接回到业务侧。

### IMD-L-01 单聊第一次问本地助手

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户 `user-001` 第一次在 IM 单聊里问本地助手 A。 |
| 前置准备 | 本地助手 A 在线；IM 业务连接已建立；该单聊之前没有会话。 |
| 操作步骤 | 通过 IM 业务统一入口发送“你好”。 |
| 预期结果 | 用户收到助手回复；后台创建一个单聊会话；第一条消息不丢；单聊历史能查到用户消息和助手最终回复。 |
| 验收重点 | 第一次聊天要经历会话创建和消息补发，最容易丢第一条消息。 |

### IMD-L-02 单聊继续问本地助手

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 同一用户在同一单聊里继续提问。 |
| 前置准备 | 已完成 IMD-L-01。 |
| 操作步骤 | 继续发送“那第二步怎么做”。 |
| 预期结果 | 系统复用原单聊会话；助手基于上下文回答；不会新建重复会话。 |
| 验收重点 | 单聊会话、助手、发送人必须匹配。 |

### IMD-L-03 单聊本地助手追问

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 本地助手在单聊里问用户补充信息。 |
| 前置准备 | 本地助手 A 能返回追问。 |
| 操作步骤 | 1. 用户发起会触发追问的问题。2. 业务侧收到追问。3. 用户提交答案。 |
| 预期结果 | 追问能展示给用户；用户答案回到正确追问；助手继续回答。 |
| 验收重点 | 如果追问来自子任务，回复必须回到子任务。 |

### IMD-L-04 单聊本地助手授权

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 本地助手在单聊里请求执行权限。 |
| 前置准备 | 本地助手 A 能返回授权请求。 |
| 操作步骤 | 分别提交允许一次、始终允许、拒绝。 |
| 预期结果 | 三种选择均被正确处理；用户能看到授权结果；非法值被拒绝。 |
| 验收重点 | 授权结果不能串到别的单聊或群聊。 |

### IMD-L-05 单聊本地助手重建会话

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 用户想重新开始，或系统发现当前上下文不可用。 |
| 前置准备 | 已有单聊会话。 |
| 操作步骤 | 触发重建，再发送新问题。 |
| 预期结果 | 旧上下文被清理；下一条消息使用新上下文；用户不会收到旧上下文重复回答。 |
| 验收重点 | 重建动作不是普通聊天问题，不能直接发给助手回答。 |

### IMD-L-06 单聊本地助手离线

| 项目 | 内容 |
| --- | --- |

| 操作步骤 | 云端发起推送。 |
| 预期结果 | 系统拒绝或忽略推送；不会乱发到其他会话。 |
| 验收重点 | 找不到会话时宁可不发，不能错发。 |

### PUSH-04 云端推送助手不匹配

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 云端推送的助手和会话中的助手不一致。 |
| 前置准备 | 构造助手账号不匹配的推送。 |
| 操作步骤 | 云端发起推送。 |
| 预期结果 | 系统拒绝发送；业务侧收不到错误助手的消息。 |
| 验收重点 | 防止一个助手把消息推到另一个助手的会话。 |

### PUSH-05 云端重复推送

| 项目 | 内容 |
| --- | --- |
| 业务场景 | 云端因为重试把同一条消息推了两次。 |
| 前置准备 | 两次推送使用同一个业务追踪标识。 |
| 操作步骤 | 连续推送两次。 |
| 预期结果 | 用户只收到一次。 |
| 验收重点 | 重复推送要去重。 |
