# 10 v1.1 Integration Checklist

> 版本：1.1
> 日期：2026-03-12

## 目标

验证 `Connection-Aligned Consumption Fix` 在真实多实例环境中的行为符合预期，重点确认：

- 实时广播已从 `session:{sessionId}` 切换到 `user-stream:{userId}`
- 非真实连接实例不会再错误消费消息
- 同一用户在多个在线端对应的实例上都能收到同一条回流消息

## 前置条件

### 部署拓扑

- `skill-server` 至少 3 个实例：A / B / C
- `gateway` 至少 3 个实例：G1 / G2 / G3
- Redis 为共享实例
- 至少 1 个在线 `pc-agent`

### 观测准备

- 打开 A / B / C 的 `skill-server` 日志
- 打开 G1 / G2 / G3 的 `gateway` 日志
- 能够区分：
  - 前端流连接落在哪个 `skill-server`
  - REST 请求命中哪个 `skill-server`
  - gateway 回流先进入哪个实例

### 建议关注字段

- `userId`
- `ak`
- `welinkSessionId`
- `toolSessionId`
- `instanceId`
- Redis channel：`user-stream:{userId}`

## Case 1：核心 A/B/C 错投回归

### 场景

- 用户前端流连接建立在 `skill-server A`
- 同一用户的发送消息 REST 请求命中 `skill-server B`
- `gateway` 回流先到 `skill-server C`

### 步骤

1. 让用户通过前端建立流连接，并确认连接在 A
2. 创建或打开一个可用会话，记录 `welinkSessionId`
3. 发送一条会触发 agent / opencode 回流的消息，并确认 REST 命中 B
4. 观察本次 invoke 报文是否带上 `userId`
5. 观察 `gateway` 是否完成 `ak -> userId` 一致性校验
6. 等待 agent / opencode 回流，并确认 gateway 回流先被 C 收到
7. 观察 C 是否将实时消息发布到 `user-stream:{userId}`
8. 观察 A 是否收到并推送消息给前端
9. 观察 B 是否没有因为“发过消息”而参与错误消费

### 期望结果

- invoke 链路中的 `userId` 校验通过
- `pc-agent` 收到的报文中不包含 `userId`
- gateway 回流后发布的 Redis channel 是 `user-stream:{userId}`，不是 `session:{sessionId}`
- A 收到实时消息并成功推给前端
- B 不会因为之前命中过 REST 而收到或消费这条回流消息
- 用户前端能看到完整返回

## Case 2：双端同时在线

### 场景

- 同一用户在两个端上同时在线
- 两个端分别连到不同的 `skill-server` 实例

### 步骤

1. 让 PC 端连接到 A
2. 让移动端连接到 B
3. 两个端打开同一个会话
4. 从任意一端发送一条会触发回流的消息
5. 观察 `user-stream:{userId}` 广播
6. 观察 A 和 B 是否都收到并向本机连接推送

### 期望结果

- A 和 B 都订阅了相同的 `user-stream:{userId}`
- 同一条实时回流消息会同时到达 A 和 B
- 两个端都能看到同一条会话返回

## Case 3：最后一条连接关闭后退订

### 场景

- 用户在某实例上的最后一条流连接关闭

### 步骤

1. 让用户只在 A 上保留一条流连接
2. 确认 A 已订阅 `user-stream:{userId}`
3. 主动关闭该连接
4. 观察 A 的退订日志
5. 再触发一条同用户的回流消息

### 期望结果

- A 在最后一条连接关闭后取消订阅 `user-stream:{userId}`
- 后续同用户回流不会再投递到 A

## Case 4：`ak -> userId` 校验失败

### 场景

- invoke 请求中携带的 `userId` 与 gateway 侧 `ak -> userId` 映射不一致

### 步骤

1. 构造或模拟一个 `userId` 不匹配的 invoke 请求
2. 观察 gateway 校验日志和响应

### 期望结果

- gateway 拒绝该次 invoke
- 不会向 `pc-agent` 下发消息
- 日志中能明确看到 `ak -> userId` 校验失败

## Case 5：旧 session 实时路径退出主链路

### 场景

- 验证主实时链路已经不再依赖 `session:{sessionId}`

### 步骤

1. 触发一条标准实时回流消息
2. 观察 `skill-server` 日志和 Redis 监控
3. 搜索是否仍出现实时 `session:{sessionId}` publish / subscribe 行为

### 期望结果

- 主实时链路只出现 `user-stream:{userId}`
- 不再出现旧的 session 级实时广播行为

## Case 6：异常重连恢复

### 场景

- 用户流连接从 A 断开后重连到 B

### 步骤

1. 让用户初始连接在 A
2. 触发一次消息并确认 A 能收到
3. 断开连接并让用户重连到 B
4. 再次触发一条回流消息
5. 观察 A / B 的订阅变化

### 期望结果

- A 完成退订
- B 完成订阅
- 重连后的实时消息只会投递到 B
