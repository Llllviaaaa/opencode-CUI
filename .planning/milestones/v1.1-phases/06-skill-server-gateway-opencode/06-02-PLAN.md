<objective>
把 `skill-server` 的实时广播从 `session:{sessionId}` 切到 `user-stream:{userId}`，并用用户流连接的首订阅 / 末退订生命周期表达“哪些实例能消费”。
</objective>

  <task type="auto" tdd="true">
    <name>Task 2: 在 SkillStreamHandler 中维护首订阅 / 末退订生命周期</name>
    <files>
      skill-server/src/main/java/com/opencode/cui/skill/ws/SkillStreamHandler.java
      skill-server/src/main/java/com/opencode/cui/skill/service/GatewayRelayService.java
    </files>
    <behavior>
      - Test 1: 某用户在本实例第一条流连接建立时，会触发订阅 `user-stream:{userId}`
      - Test 2: 某用户在本实例最后一条流连接关闭时，才会取消订阅 `user-stream:{userId}`
      - Test 3: 同一实例内同一用户的多条连接并存时，不会因为关闭其中一条就误退订
    </behavior>
    <action>在 `SkillStreamHandler` 中为每个 `userId` 维护本机 `activeConnectionCount`，并在 `afterConnectionEstablished`、`afterConnectionClosed`、`handleTransportError`、清理逻辑中统一驱动订阅生命周期；只在计数从 `0 -> 1` 时订阅，只在 `1 -> 0` 时取消订阅。保持 TTL 仅作为异常断开后的兜底清理，不把 TTL 变成正常消费判定条件。</action>
    <verify>
      <automated>mvn -pl skill-server -DskipTests compile</automated>
    </verify>
    <done>实时消费资格完全由 user-stream 订阅关系表达，且同实例多连接场景不会误退订。</done>
  </task>

  <task type="auto">

    <files>

    <action>整理 `GatewayRelayService` 与 `SkillStreamHandler` 的职责边界：实例一旦通过 `user-stream:{userId}` 收到消息，就直接向本机该用户的活动流连接推送，而不再额外查 owner 集合；补充幂等防护和日志，避免一个实例重复订阅同一用户导致重复推送。</action>
    <verify>

  </task>

<must_haves>
- 消费资格由 `user-stream:{userId}` 的订阅关系天然表达，不再依赖额外 owner 集合查询。
- 同一用户可以在多个实例上同时有有效流连接，并都能收到同一会话的 AI 回流消息。
- 本机多连接计数必须可靠，避免误退订造成丢消息。
</must_haves>

<success_criteria>
- 回流消息按 `userId` 广播到 `user-stream:{userId}`
- 只有真正持有该用户流连接的 skill-server 实例会订阅并收到消息
- 收到 user-stream 消息的实例能稳定推送本机连接，不再被 REST 落点实例污染
</success_criteria>
