<objective>
为 `skill-server -> gateway -> pc-agent -> gateway -> skill-server` 链路补齐可信 `userId` 上下文，确保 invoke 侧可校验、回流侧可补全，同时不把 `userId` 扩散到 `pc-agent`。
</objective>

<tasks>
  <task type="auto" tdd="true">
    <name>Task 1: 扩展 GatewayMessage 的 userId 路由上下文字段</name>
    <files>
      ai-gateway/src/main/java/com/opencode/cui/gateway/model/GatewayMessage.java
      skill-server/src/main/java/com/opencode/cui/skill/service/GatewayRelayService.java
      skill-server/src/main/java/com/opencode/cui/skill/controller/SkillMessageController.java
    </files>
    <behavior>
      - Test 1: invoke 报文从 `skill-server` 发往 `gateway` 时会显式携带 cookie 解析出的 `userId`
      - Test 2: 发给 `pc-agent` 的报文仍不依赖 agent 自己携带 `userId`
    </behavior>
    <action>在 `GatewayMessage` 顶层增加 `userId` 并补齐 builder、copy 或 factory 入口；在 `SkillMessageController` 与 `GatewayRelayService.sendInvokeToGateway(...)` 的调用链上从现有 cookie / access control 上下文提取可信 `userId`，把它写入发往 gateway 的 invoke 消息。不要把 `userId` 下沉到 payload 结构，也不要要求 `pc-agent` 改协议感知该字段。</action>
    <verify>
      <automated>mvn -pl ai-gateway,skill-server -DskipTests compile</automated>
    </verify>
    <done>`GatewayMessage` 能承载 `userId`，并且 `skill-server` 发往 `gateway` 的 invoke 路径已经稳定写入该字段。</done>
  </task>

  <task type="auto" tdd="true">
    <name>Task 2: 在 gateway 落地 ak 到 userId 的校验与回流补全</name>
    <files>
      ai-gateway/src/main/java/com/opencode/cui/gateway/service/SkillRelayService.java
      ai-gateway/src/main/java/com/opencode/cui/gateway/service/RedisMessageBroker.java
      ai-gateway/src/main/java/com/opencode/cui/gateway/model/GatewayMessage.java
    </files>
    <behavior>
      - Test 1: gateway 收到 invoke 时，如果报文 `userId` 与 `ak -> userId` 映射不一致，会拒绝继续调用 agent
      - Test 2: agent 回流时 gateway 能基于 `ak -> userId` 把可信 `userId` 补回 `GatewayMessage`
    </behavior>
    <action>复用 gateway 现有 Redis 协调能力，把 `ak -> userId` 视为可信映射来源；在 invoke 入口增加一致性校验，校验失败返回明确错误并记录日志；在 `pc-agent` 回流消息重新封装为 `GatewayMessage` 时，基于 `ak` 查询并补回 `userId`。避免把消费资格判断也前置到 gateway，只负责校验与补全。</action>
    <verify>
      <automated>mvn -pl ai-gateway -DskipTests compile</automated>
    </verify>
    <done>gateway 对 invoke 执行 `ak -> userId` 校验，且回流给 `skill-server` 的消息已经带有可信 `userId`。</done>
  </task>

  <task type="auto">
    <name>Task 3: 为 userId 上下文补充日志与兼容性保护</name>
    <files>

    <verify>
      <automated>mvn -pl ai-gateway,skill-server -DskipTests compile</automated>
    </verify>
    <done>关键链路具备可追踪日志，缺失或校验失败的 `userId` 不会被静默放过。</done>

<must_haves>
- invoke 请求经过非连接实例时，后续 AI 回流消息的消费归属不能被该实例改变。
- `userId` 必须由服务端生成、校验和补全，`pc-agent` 不承担任何用户身份语义。
- Redis key、日志字段与报文字段命名保持一致，避免联调时出现隐式契约漂移。
</must_haves>

<success_criteria>
- `skill-server` 发给 `gateway` 的 invoke 消息带有可信 `userId`
- gateway 能校验 `ak -> userId` 一致性并在回流时补回 `userId`
- 缺失或不一致的 `userId` 会被明确拒绝，而不是落回旧路径
</success_criteria>
